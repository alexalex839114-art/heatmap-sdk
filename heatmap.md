Ты — Senior Python Engineer с опытом работы с биржевыми API (Binance, Bybit), высокопроизводительной обработкой данных (NumPy, Pandas, Parquet), веб-фреймворками (FastAPI) и интерактивной визуализацией (OpenSeadragon, Canvas).



Твоя задача: спроектировать и написать полную систему реального времени для визуализации стакана (тепловая карта, как Bookmap) для Binance и Bybit. Система работает локально (один пользователь), собирает данные через WebSocket, хранит последние 2 дня в Parquet, отдаёт через веб-интерфейс тепловую карту с возможностью зумить/панорамировать (Deep Zoom), обновляющуюся в реальном времени.



\## Требования к системе



\### 1. Сбор данных (WebSocket бот)



\- Одновременная поддержка Binance и Bybit (можно переключаться в UI или собирать отдельно, на выбор пользователя).

\- Подписка на поток частичных обновлений стакана (depth) для выбранной пары (например, BTCUSDT). Глубина обновлений от биржи — максимум 5000 уровней.

\- В памяти поддерживается полный order book (словари bid и ask, сортированные). Применяются все дельты (добавление, изменение, удаление). Цены хранятся как Decimal (для точности).

\- Каждые 100 мс (по таймеру) берётся снимок текущего состояния стакана и сохраняется в буфер. Снимок содержит:

&#x20; - timestamp (int64, ns)

&#x20; - bid\_prices: list\[float] (первые N уровней, N = глубина хранения, фиксированная 1000)

&#x20; - bid\_volumes: list\[float]

&#x20; - ask\_prices: list\[float] (первые 1000 уровней)

&#x20; - ask\_volumes: list\[float]

&#x20; - trade\_price: float или None (если за последние 100 мс была сделка — берётся последняя)

&#x20; - trade\_volume: float или 0

\- Если за 100 мс не было никаких изменений — повторяется предыдущий снимок (интерполяция).

\- Сборщик должен работать в отдельном потоке или асинхронно, не блокируя веб-сервер.



\### 2. Хранение (Parquet, скользящие 2 дня)



\- Данные сохраняются в Parquet-файлы, партиционированные по паре, дате и часу.

\- Путь: `data/{exchange}/{pair}/year=YYYY/month=MM/day=DD/hour=HH.parquet`.

\- Схема Parquet: строго соответствует снимку (все колонки: timestamp, bid\_prices, bid\_volumes, ask\_prices, ask\_volumes, trade\_price, trade\_volume). Массивы хранить как `list<float64>` фиксированной длины (1000). Использовать `pyarrow` или `fastparquet`.

\- Каждый час создаётся новый файл. При старте системы загружаются последние 2 дня данных из Parquet (для восстановления состояния после перезапуска). Старые файлы (старше 2 дней) автоматически удаляются.

\- Если данных за предыдущие дни нет (первый запуск), система работает без истории и начинает заполнять текущий час.



\### 3. Генерация тепловой карты (бекенд)



\- Клиент (браузер) запрашивает \*\*уже сбинженные данные\*\* для отображения, а не сырые снимки.

\- Эндпоинт `POST /api/heatmap/history` принимает:

&#x20; - `pair`: str (например "BTCUSDT")

&#x20; - `exchange`: "binance" или "bybit"

&#x20; - `depth`: int (глубина отображения, не более 1000, пользователь выбирает в UI)

&#x20; - `start\_time`, `end\_time`: ISO timestamps (за последние 2 дня)

&#x20; - `bin\_ms`: int (агрегация по времени в миллисекундах, по умолчанию = 100 мс? Но для истории, чтобы не было слишком широко, можно динамически: например, итоговая ширина не более 12000 пикселей, тогда bin\_ms = (end - start) / 12000)

&#x20; - `trade\_aggregation\_ms`: int (агрегация сделок по времени, по умолчанию 1000 мс)



\- Сервер:

&#x20; 1. Читает Parquet за запрошенный период.

&#x20; 2. Выполняет биннинг (усреднение) по временным слотам размером `bin\_ms`. Для каждого слота:

&#x20;    - берёт все снимки внутри интервала;

&#x20;    - вычисляет для каждого уровня цены (от 1 до `depth`) средний log1p(объём) (логарифмическая нормализация по всем данным глобально — но проще по запрошенному периоду, так быстрее);

&#x20;    - mid-price = (лучший bid + лучший ask) / 2 (усреднённый по слотам);

&#x20;    - сделки: суммирует объём по всем сделкам в интервале (или агрегирует по `trade\_aggregation\_ms` и выдаёт список точек).

&#x20; 3. Возвращает JSON:

&#x20;    - `time\_bins`: массив Unix ms для каждого столбца.

&#x20;    - `price\_levels`: массив цен (ось Y) – строится как объединённый диапазон от минимальной bid цены до максимальной ask цены за весь период, с линейным шагом (количество уровней = высота карты, например 800).

&#x20;    - `absolute\_grid`: 2D массив (высота × ширина) нормализованной яркости (uint8) для абсолютной карты.

&#x20;    - `relative\_grid`: 2D массив для относительной карты (цена относительно mid-price, выраженная в спредах).

&#x20;    - `trades`: список { time\_index, price\_level\_index, volume\_log }.



\- Эндпоинт `GET /api/heatmap/current` возвращает последний слот (для инициализации).



\### 4. Реальное время (WebSocket между сервером и клиентом)



\- Сервер открывает WebSocket endpoint `/ws/live`.

\- Когда бот-сборщик формирует новый снимок (каждые 100 мс), сервер:

&#x20; 1. Агрегирует его до текущего выбранного пользователем `depth` (динамически, потому что глубина задаётся в UI и может измениться).

&#x20; 2. Преобразует в один столбец тепловой карты (массив яркостей длины = высота карты, предварительно известной клиенту).

&#x20; 3. Отправляет клиенту сообщение: `{ "col\_index": номер\_столбца, "absolute\_column": \[uint8], "relative\_column": \[uint8], "trade": { "price\_y": int, "volume\_log": float } или null }`.

\- Клиент получает сообщение и рисует новый столбец справа от существующей карты, сдвигая окно (как в Bookmap). Максимальная ширина в пикселях ограничена (например, 4000 столбцов), старые столбцы отбрасываются.



\### 5. Веб-интерфейс (HTML/CSS/JS, OpenSeadragon)



\- Страница содержит:

&#x20; - Панель управления: выбор биржи (Binance/Bybit), ввод пары (например, BTCUSDT), поле `Depth` (число, 1-1000), поле `Trade aggregation (ms)` (выбор 100, 500, 1000, 5000), кнопка "Start".

&#x20; - Два контейнера для тепловых карт (верхняя абсолютная, нижняя относительная). Каждый контейнер — это элемент OpenSeadragon, загружающий .dzi тайлы для исторической части.

&#x20; - Поверх OpenSeadragon — Canvas слой для рисования реального времени (новые столбцы рисуются поверх тайлов, потому что Deep Zoom не поддерживает динамическое добавление).

&#x20; - Легенда: цвет/яркость = log объема.



\- При нажатии "Start":

&#x20; 1. Клиент запрашивает историю за последние 2 дня через `/api/heatmap/history` (с выбранными параметрами). Сервер генерирует тайлы Deep Zoom (асинхронно, если их нет в кеше) и возвращает URL .dzi.

&#x20; 2. OpenSeadragon загружает тайлы.

&#x20; 3. Устанавливается WebSocket соединение `/ws/live`.

&#x20; 4. При получении новых столбцов: клиент получает текущую позицию прокрутки, рисует новый столбец на Canvas (поверх OpenSeadragon). Если ширина превышает лимит, удаляется самый левый столбец.

&#x20; 5. Обработка выбора глубины: при изменении глубины нужно перезапросить историю и переподключиться с новыми параметрами.



\- Сделки: отображаются как точки на абсолютной карте (цвет белый, радиус = sqrt(volume\_log) или пропорционально). При агрегации сделок за интервал времени, точка может быть крупнее.



\### 6. Генерация Deep Zoom тайлов (фоновый процесс)



\- Для каждого запроса истории (уникальная комбинация pair, exchange, depth, bin\_ms, start, end) сервер должен сгенерировать тайлы один раз и сохранить в кеш (папка `tiles/{hash}/`). Использовать Pillow для нарезки PNG (исходная ширина = количество бин-столбцов, высота = высота карты, фиксированная 800-1000 пикселей). Генерировать .dzi манифест.

\- Для ускорения: тайлы генерируются в фоне, а клиент сначала получает ответ, что они готовятся, и через polling получает URL. Но для локального использования можно генерировать синхронно (если история не огромная).

\### 9. Упрощения для первой версии (чтобы не усложнять)



\- Не поддерживать одновременную работу с несколькими парами. Одна пара за раз.

\- При изменении параметров (глубина, агрегация) — полная перезагрузка страницы (проще, чем динамически перестраивать).

\- Не делать сложную интерполяцию истории, только повтор последнего снимка.

\- Для относительной карты: для каждого слота вычисляем массив отклонений: `(price - mid\_price) / spread`, обрезаем до ±3 спреда и нормализуем в \[0,255].
